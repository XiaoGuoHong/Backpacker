# SmartVoyage 智慧旅行助手系统需求分析

## 1. 文档说明

### 1.1 编写目的

本文档面向 SmartVoyage 开发团队，用于明确首期最小可行产品（MVP）的建设目标、业务范围、功能需求、数据与接口约束、非功能要求以及验收标准，并作为后续系统设计、任务拆分、开发和测试的依据。

### 1.2 项目阶段

本文档描述 SmartVoyage 首期 MVP。MVP 的目标是打通“用户自然语言提问—系统识别意图—调用查询能力—整合并返回结果”的完整链路，验证分布式 AI 架构在旅行信息整合场景中的可行性。

### 1.3 术语

| 术语 | 说明 |
| --- | --- |
| LLM | 大语言模型，用于理解用户问题、提取查询参数、选择工具和组织回答。 |
| MCP | Model Context Protocol，用于规范大语言模型与数据库、API、爬虫等外部工具之间的连接。 |
| A2A | Agent-to-Agent 协议，用于不同智能代理之间的任务分派、结果传递和状态协作。 |
| Agent | 负责某一类任务的智能代理，如天气查询代理、交通票务查询代理。 |
| MVP | 最小可行产品，即能够完成核心业务闭环并用于验证产品价值的首期版本。 |

## 2. 项目背景

旅行规划通常涉及天气、火车票、飞机票和演出活动等多类信息。传统旅行工具要求用户在多个应用或网站之间反复切换，分别输入地点、日期和其他条件，存在信息分散、操作重复、结果格式不统一和时间成本较高等问题。

SmartVoyage 是一个智慧旅行助手系统，计划基于 LLM、MCP 和 A2A 构建统一的自然语言查询入口，并通过数据库、第三方接口、合规爬虫和微服务整合旅行相关信息。例如，用户输入“北京 2025-08-11 天气”，系统识别查询类型、地点和日期，调用相应数据能力，并返回结构化、可理解的结果。

项目当前处于需求分析阶段。本文中的功能均为待开发需求，不代表已经实现或已经接入真实数据源。

## 3. 建设目标

### 3.1 总体目标

构建一个统一的旅行信息查询入口，使用户能够通过自然语言完成天气、交通票务和演唱会票务查询，减少在不同平台之间切换和重复输入信息的成本。

### 3.2 MVP 目标

1. 支持天气、火车票、飞机票和演唱会票务信息查询。
2. 支持从自然语言中识别查询意图，并提取城市、日期、出发地、目的地等关键参数。
3. 通过 MCP 统一调用数据库、第三方 API 或合规爬虫提供的查询工具。
4. 通过必要的专业 Agent 完成任务分派与结果汇总，并验证 A2A 协作链路。
5. 将不同来源的结果转换为统一、清晰且带有数据来源与更新时间的响应。
6. 对参数缺失、无查询结果、数据源异常和超时等情况提供明确反馈。

### 3.3 非目标

以下内容不属于首期 MVP：

- 票务下单、支付、抢票、退票和改签；
- 酒店、景点、餐饮和租车预订；
- 自动生成完整的多日行程；
- 个性化用户画像和长期偏好学习；
- 会员、积分、营销和商业结算；
- 对第三方平台账号的自动登录或代操作；
- 面向生产环境的大规模高并发和多地域部署。

## 4. 用户与使用场景

### 4.1 目标用户

- 需要快速了解目的地天气的旅行者；
- 需要比较火车或飞机出行信息的用户；
- 希望查询目的地演唱会信息的用户；
- 需要通过统一入口收集旅行基础信息的行程规划者。

首期 MVP 不要求用户注册或登录，以降低核心查询链路的实现复杂度。

### 4.2 核心使用场景

#### 场景一：天气查询

用户输入“北京 2025-08-11 天气怎么样”。系统识别城市和日期，查询天气数据，并返回天气状况、温度范围、降水概率等可获得字段。

#### 场景二：火车票查询

用户输入“8 月 11 日从北京到上海的火车”。系统识别出发地、目的地和日期，返回符合条件的车次、出发到达时间、历时、席别和余票等可获得字段。

#### 场景三：飞机票查询

用户输入“8 月 11 日北京飞上海的航班”。系统识别航线和日期，返回航班号、机场、起降时间、价格和舱位等可获得字段。

#### 场景四：演唱会查询

用户输入“8 月北京有哪些演唱会”。系统识别城市和时间范围，返回演出名称、艺人、时间、场馆、票价区间、售票状态和信息来源等可获得字段。

#### 场景五：信息不完整

用户输入“帮我查一下火车票”，但未提供出发地、目的地或日期。系统不应自行虚构条件，而应指出缺少的必要参数并引导用户补充。

## 5. MVP 业务范围

### 5.1 查询类型

| 查询类型 | 必要参数 | 可选参数 | 主要输出 |
| --- | --- | --- | --- |
| 天气 | 城市、日期 | 区县、时间段 | 天气状况、温度、降水、风力及数据更新时间 |
| 火车票 | 出发地、目的地、日期 | 出发时间段、车次类型、席别 | 车次、站点、时间、历时、席别、余票及来源 |
| 飞机票 | 出发地、目的地、日期 | 时间段、舱位、是否直飞 | 航班、机场、时间、价格、舱位及来源 |
| 演唱会 | 城市、日期或时间范围 | 艺人、场馆、价格范围 | 演出、艺人、时间、场馆、票价、状态及来源 |

### 5.2 查询模式

MVP 支持单轮查询和一次必要的参数补充。系统应保留当前会话中已确认的查询参数，使用户能够通过“改成上海”“日期换成 8 月 12 日”等简短表达修正当前查询。MVP 不要求长期保存跨会话上下文。

### 5.3 数据使用原则

1. 优先使用合法、稳定、字段清晰的官方 API 或已授权数据接口。
2. 数据库可用于缓存、演示数据或内部结构化数据查询。
3. 使用爬虫时必须遵守目标网站的服务条款、robots 规则、访问频率限制和适用法律，不绕过验证码、登录验证或反爬机制。
4. 查询结果必须标识数据来源和最近更新时间；无法确认实时性的结果不得宣称为实时数据。
5. 某类真实数据源未接入时，可使用明确标记的模拟数据完成开发和测试，但不得在用户界面中伪装为真实查询结果。

## 6. 总体业务流程

1. 用户输入自然语言问题。
2. 接入服务创建请求标识，完成基础格式校验和安全检查。
3. LLM 识别查询意图并提取结构化参数。
4. 系统校验必要参数；参数不完整时向用户发起补充询问。
5. 调度 Agent 根据查询类型选择对应的专业查询 Agent。
6. 专业 Agent 通过 MCP 调用数据库、第三方 API 或合规爬虫工具。
7. 工具返回结构化结果、来源、更新时间或错误信息。
8. 专业 Agent 对结果进行规范化，调度 Agent 汇总结果。
9. LLM 在不改变事实数据的前提下生成用户可读回答。
10. 系统记录不含敏感信息的请求状态、耗时和错误类型，并向用户返回结果。

## 7. 功能需求

### 7.1 自然语言交互

#### FR-01 查询输入

- 系统应支持用户以中文自然语言提交查询。
- 系统应拒绝空输入，并对超出长度限制的输入给出提示。
- 系统应为每次查询生成唯一请求标识，便于链路追踪。

#### FR-02 意图识别

- 系统应识别天气、火车票、飞机票和演唱会四类意图。
- 无法确定意图时，系统应要求用户说明查询目标，不得随机选择工具。
- 同一输入包含多个独立查询意图时，MVP 应提示用户拆分问题或仅在明确支持的情况下按顺序处理，不保证跨领域联合规划。

#### FR-03 参数提取与标准化

- 系统应提取各查询类型所需的必要参数和可选参数。
- 系统应将自然语言日期转换为统一日期格式，并保留原始表达用于追踪。
- 对“今天”“明天”等相对日期，系统应基于服务端配置的业务时区计算，并在响应中显示转换后的绝对日期。
- 对存在歧义的城市、车站、机场或日期，系统应向用户确认，不得擅自猜测。
- 系统应校验日期范围、地点格式和参数组合的合理性。

#### FR-04 参数补充与修改

- 必要参数缺失时，系统应一次性列出当前缺失的必要参数。
- 用户补充或修改参数后，系统应合并当前会话中已确认的信息并重新查询。
- 切换查询类型时，系统不应错误沿用不相关参数。

### 7.2 天气查询

#### FR-05 天气数据查询

- 系统应按城市和日期查询天气信息。
- 返回字段应至少包含城市、日期、天气状况和数据来源；温度、降水概率、风力等字段按数据源实际能力返回。
- 查询日期超出数据源支持范围时，应明确说明可查询范围。
- 无结果时，应区分地点无法识别、日期不可用和数据源无数据。

### 7.3 火车票查询

#### FR-06 火车票信息查询

- 系统应按出发地、目的地和日期查询车次信息。
- 结果应按统一结构返回车次、出发站、到达站、出发时间、到达时间和历时。
- 席别、余票和价格仅在数据源提供时显示，并标注查询时间。
- 系统应说明结果仅供信息参考，最终库存和价格以售票平台为准。

### 7.4 飞机票查询

#### FR-07 航班信息查询

- 系统应按出发地、目的地和日期查询航班信息。
- 结果应按统一结构返回航班号、出发机场、到达机场、起飞时间和到达时间。
- 价格、舱位、经停和准点信息仅在数据源提供时显示，并标注币种和查询时间。
- 系统应说明票价和舱位可能变化，最终信息以承运方或售票平台为准。

### 7.5 演唱会查询

#### FR-08 演唱会信息查询

- 系统应按城市及日期或时间范围查询演唱会信息。
- 结果应按统一结构返回演出名称、艺人、日期、场馆和信息来源。
- 票价区间、开售时间、售票状态和购票页面仅在数据源提供时显示。
- 系统不得宣称可以代购或保证票务库存。

### 7.6 Agent 调度与 A2A 协作

#### FR-09 任务路由

- 系统应设置一个负责意图路由与结果汇总的调度 Agent。
- 系统应按业务需要设置天气、交通票务和演唱会查询 Agent；火车与飞机可在交通票务 Agent 内作为独立工具处理。
- Agent 之间传递的数据应采用明确的结构化协议，至少包含请求标识、任务类型、参数、状态、结果或错误。
- 每个任务应具备超时控制，避免单个 Agent 或工具无限等待。

#### FR-10 结果汇总

- 调度 Agent 应校验专业 Agent 返回结果的状态和基本字段。
- LLM 只能基于工具实际返回的数据组织回答，不得补造票价、余票、天气或演出信息。
- 部分数据源失败时，系统应返回已获得的有效结果，并说明缺失部分及原因。

### 7.7 MCP 工具调用

#### FR-11 工具注册与发现

- MCP 工具应声明工具名称、用途、输入参数、输出结构、超时和错误类型。
- Agent 只能调用已注册且被授权的工具。
- 工具输入必须经过服务端校验，不能直接信任 LLM 生成的参数。

#### FR-12 数据库查询

- 数据库查询仅允许使用只读账号和预先授权的数据范围。
- 若采用 LLM 生成 SQL，系统必须限制为单条只读查询，禁止数据定义、数据修改、多语句执行和任意系统表访问。
- 执行 SQL 前应进行语法、操作类型、表字段白名单和结果数量检查，并设置查询超时。
- 推荐优先使用参数化查询或受控查询模板；自然语言生成 SQL 仅用于结构未知或查询组合确有必要的受控场景。
- 系统不得在响应、日志或错误详情中暴露数据库凭据、内部连接信息或完整 SQL 异常堆栈。

### 7.8 结果展示

#### FR-13 统一响应

- 响应应包含查询条件摘要、结果主体、数据来源、查询时间和必要提示。
- 多条结果应使用列表或表格等易读形式展示，并支持合理的结果数量限制。
- 系统应区分“没有符合条件的数据”和“查询服务暂时不可用”。
- 当结果包含跳转链接时，只能返回数据源提供或系统配置的可信链接。

### 7.9 异常处理

#### FR-14 用户输入异常

- 输入缺失、格式错误或存在歧义时，系统应给出可操作的修正提示。
- 用户请求超出 MVP 范围时，系统应说明当前支持的查询类型。

#### FR-15 外部服务异常

- 工具超时、限流、鉴权失败、数据格式变化或服务不可用时，应转换为统一错误类型。
- 面向用户的错误信息不得包含密钥、Token、Cookie、内部地址、调用堆栈或供应商敏感响应。
- 对适合重试的瞬时错误可进行有限次数重试；重试次数和退避策略由技术设计确定。

## 8. 数据需求

### 8.1 核心数据对象

| 数据对象 | 核心字段 |
| --- | --- |
| 查询请求 | 请求标识、会话标识、原始输入、意图、结构化参数、创建时间 |
| 查询任务 | 任务标识、请求标识、Agent 类型、工具类型、状态、开始和结束时间 |
| 工具结果 | 数据内容、来源、来源更新时间、抓取或查询时间、状态、错误类型 |
| 统一响应 | 请求标识、查询条件摘要、规范化结果、来源、生成时间、提示信息 |

### 8.2 数据状态

查询任务至少应支持以下状态：待处理、处理中、成功、部分成功、失败和超时。状态名称可在技术设计阶段映射为统一枚举，但业务含义必须保持一致。

### 8.3 数据保留

- MVP 默认只保留排障和性能分析所需的最少请求元数据。
- 原始用户输入和查询结果的保存周期应可配置。
- 日志和持久化数据中不得记录密码、API Key、Access Token、Cookie、Authorization 请求头等敏感信息。
- 若未来引入账号或个人信息，应另行进行隐私影响分析并更新需求。

## 9. 外部接口需求

### 9.1 客户端接口

MVP 至少提供一种可用的用户交互入口，可为 Web 页面或 API 调试界面。服务端应提供统一查询接口，并支持返回查询结果、参数补充提示和标准错误信息。

### 9.2 Agent 接口

Agent 间请求至少应包含：

- 请求标识和任务标识；
- 任务类型和版本；
- 已校验的结构化参数；
- 调用截止时间或超时配置；
- 调用链追踪信息。

Agent 间响应至少应包含：

- 任务状态；
- 结构化结果；
- 数据来源和时间信息；
- 标准错误码及可公开的错误说明。

### 9.3 MCP 工具接口

所有 MCP 工具必须采用结构化输入和输出，明确必要字段、可选字段和字段类型。工具异常不得以未处理的自由文本直接传递至最终用户。

### 9.4 第三方接口

具体天气、铁路、航班和演唱会数据供应商在技术调研后确定。接入前必须确认接口授权、调用配额、数据更新频率、费用、稳定性和使用条款。供应商未确定不影响使用模拟工具完成 MVP 主链路开发。

## 10. 非功能需求

### 10.1 性能

- 在外部数据源正常的情况下，普通单类查询的服务端端到端响应时间目标为 10 秒以内。
- 超过 3 秒仍未完成的查询，客户端应显示处理中状态或等待提示。
- 单次返回结果默认不超过 20 条，防止响应过大；具体分页方案在接口设计阶段确定。
- 每个外部工具必须设置独立超时，不得无限等待。

### 10.2 可用性与容错

- 单个外部数据源失败不应导致整个服务进程崩溃。
- 系统应对超时、限流和临时网络错误进行统一处理。
- 服务应提供基础健康检查，能够区分应用存活状态与关键依赖可用状态。

### 10.3 安全

- 密钥、Token 和数据库凭据必须通过环境变量或安全配置系统注入，不得写入源码、测试数据或文档。
- 外部输入、LLM 输出和工具返回值均应视为不可信数据并进行校验。
- 数据库使用最小权限只读账号，限制可访问表、字段、结果数量和执行时间。
- 防止提示词注入诱导系统泄露系统提示、凭据、内部工具定义或执行越权操作。
- 对外部 URL、SQL、日志内容和页面输出进行必要的注入与转义防护。
- 对接口实施基础限流和请求大小限制。

### 10.4 可观测性

- 日志应包含请求标识、任务状态、Agent 和工具类型、耗时以及标准错误码。
- 日志不得记录真实密钥、完整认证信息或不必要的个人数据。
- 应统计各查询类型的调用次数、成功率、超时率和响应时间。
- 应能够根据请求标识追踪一次查询涉及的 Agent 和工具调用链。

### 10.5 可维护性

- 天气、火车、飞机和演唱会数据适配应与核心调度逻辑解耦。
- 更换数据供应商时，不应修改用户查询流程和统一响应模型。
- Agent、MCP 工具和错误码应有清晰的版本和接口约定。
- 开发、测试和演示环境的配置应相互隔离。

### 10.6 兼容性

- MVP 的 Web 入口如被采用，应支持主流桌面浏览器的近期稳定版本。
- 日期、时间和时区在接口中采用统一格式；面向用户显示时明确业务时区。
- 价格字段必须同时包含数值和币种，禁止只返回无单位数字。

## 11. 业务规则

1. 系统只提供信息查询，不构成出票、预订或购买承诺。
2. 余票、票价、天气和售票状态具有时效性，必须展示查询时间和来源。
3. LLM 不作为事实数据源，所有事实性结果必须来自已调用的工具或数据库。
4. 未取得的数据字段应省略或明确标记为暂无数据，不得由模型推断补全。
5. 用户条件存在歧义时，系统必须先确认再查询。
6. 查询日期超出数据源能力时，应说明限制，不得返回伪造预测。
7. 模拟数据必须在系统响应中清楚标记为演示数据。

## 12. 验收标准

### 12.1 核心链路验收

| 编号 | 验收场景 | 通过标准 |
| --- | --- | --- |
| AC-01 | 天气查询 | 输入包含有效城市和日期的天气问题后，系统正确识别意图与参数，并返回带来源和查询时间的天气结果。 |
| AC-02 | 火车票查询 | 输入有效出发地、目的地和日期后，系统返回结构化车次信息；数据源不提供余票时不虚构余票。 |
| AC-03 | 飞机票查询 | 输入有效航线和日期后，系统返回结构化航班信息，并正确显示机场、时间及可获得的价格信息。 |
| AC-04 | 演唱会查询 | 输入有效城市和时间范围后，系统返回结构化演出信息或明确的无结果提示。 |
| AC-05 | 参数缺失 | 输入缺少必要参数的查询后，系统明确列出缺失项且不调用不必要的数据工具。 |
| AC-06 | 参数修改 | 用户在当前会话修改地点或日期后，系统使用修改后的完整条件重新查询。 |
| AC-07 | 意图不明确 | 输入无法归类时，系统请求澄清，不随机返回某类查询结果。 |
| AC-08 | 数据源无结果 | 系统明确返回无符合条件的数据，并与系统异常相区分。 |
| AC-09 | 工具超时 | 任一工具超时后，请求在规定时间内结束，并返回可理解且不泄露内部信息的提示。 |
| AC-10 | 部分失败 | 多工具场景中一个工具失败时，系统保留有效结果并说明缺失部分。 |
| AC-11 | SQL 安全 | 非只读、多语句、越权表字段或超量查询被执行前拦截，并留下脱敏审计记录。 |
| AC-12 | 敏感信息保护 | 用户响应、应用日志和异常信息中不出现 API Key、Token、Cookie、数据库凭据或认证请求头。 |
| AC-13 | 结果真实性 | 抽查响应中的事实字段均能追溯至本次工具返回，模型未自行补造事实数据。 |
| AC-14 | 链路追踪 | 可使用请求标识定位该次查询的意图识别、Agent 调度、工具调用状态和耗时。 |

### 12.2 测试要求

- 每类查询至少覆盖正常、参数缺失、无结果、数据格式异常和工具超时场景。
- 意图识别和参数提取应建立固定测试样例集，覆盖不同中文表达、相对日期和地点歧义。
- MCP 工具和 Agent 接口应具备契约测试。
- SQL 安全限制、提示词注入防护和敏感信息脱敏应具备专项测试。
- 核心查询链路应具备端到端测试；外部服务不稳定时应使用可控模拟服务完成自动化验证。

## 13. MVP 优先级

### P0：必须完成

- 统一自然语言查询入口；
- 四类查询意图识别与参数提取；
- 必要参数校验和补充询问；
- 调度 Agent 与专业查询 Agent 的基本 A2A 协作；
- MCP 工具注册、调用、超时和标准错误处理；
- 每类查询至少一个可运行的数据适配器或明确标记的模拟工具；
- 统一响应、数据来源和查询时间展示；
- SQL 只读控制、敏感信息保护和基础链路日志；
- 核心自动化测试和演示流程。

### P1：MVP 稳定性增强

- 当前会话内的查询条件修改；
- 部分失败结果返回；
- 结果排序、筛选和数量限制；
- 指标统计、依赖健康检查和更完整的契约测试；
- 在授权条件允许时，将模拟数据适配器替换为真实数据源。

## 14. 后续扩展方向

MVP 验证通过后，可根据用户反馈和数据授权情况评估以下能力：

- 酒店、景点、餐饮和本地交通查询；
- 多来源比价和跨领域行程组合；
- 多日行程生成与动态调整；
- 用户账号、收藏、历史记录和偏好管理；
- 票务平台跳转、预订和支付；
- 语音交互、多语言和移动端；
- 更细粒度的 Agent 协作、异步任务与人工确认机制。

上述内容均不属于当前 MVP 的交付承诺，进入开发前需另行完成需求评审。

## 15. 风险与待技术调研事项

### 15.1 主要风险

| 风险 | 影响 | MVP 应对措施 |
| --- | --- | --- |
| 数据接口授权或稳定性不足 | 无法持续提供真实数据 | 抽象数据适配层，先用合规模拟工具验证主链路，再逐项替换。 |
| 票价、余票和演出状态变化快 | 展示结果可能过期 | 展示来源和查询时间，增加时效性提示，不承诺库存。 |
| LLM 意图或参数识别错误 | 调错工具或返回错误条件 | 结构化输出、参数校验、歧义确认和固定样例测试。 |
| LLM 生成不真实内容 | 损害结果可信度 | 要求事实字段仅来自工具结果，并在输出前进行字段约束。 |
| 自然语言生成 SQL 越权 | 数据泄露或数据库受损 | 只读账号、白名单、解析校验、查询限制、超时和审计。 |
| 多 Agent 增加链路复杂度 | 延迟升高、排障困难 | MVP 仅保留必要 Agent，统一协议、错误码和链路追踪。 |
| 爬虫合规和页面变化 | 接口中断或产生合规风险 | 优先官方或授权接口；限制频率，不绕过访问控制。 |

### 15.2 开发前需确认的技术事项

以下事项由开发团队在技术选型和架构设计阶段形成明确结论，不影响本需求文档对业务边界的定义：

1. 各类数据源的供应商、授权方式、费用、配额和更新频率。
2. LLM 供应商、模型能力、调用成本以及不可用时的降级方式。
3. MCP Server 的部署方式、工具清单和认证机制。
4. A2A 协议版本、Agent 消息结构、同步或异步调用方式。
5. 数据库类型、允许查询的表字段以及模拟数据结构。
6. MVP 用户入口采用 Web 页面、API 调试界面或两者同时提供。
7. 业务时区、可查询日期范围、性能基线和部署环境容量。

## 16. 需求完成定义

当以下条件全部满足时，可认为 SmartVoyage MVP 达到需求层面的完成标准：

1. P0 功能全部实现并通过对应验收场景。
2. 四类查询均能使用真实或明确标记的模拟数据完成端到端演示。
3. 参数缺失、无结果、工具超时和数据源异常均有可验证的处理路径。
4. 所有事实性结果可追溯到工具或数据库返回。
5. SQL 只读限制和敏感信息保护通过专项测试。
6. 开发团队已形成接口文档、部署说明、测试报告和已知限制清单。
