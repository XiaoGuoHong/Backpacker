# SmartVoyage 智慧旅行助手系统技术设计文档

## 1. 文档说明

### 1.1 编写目的

本文档在《SmartVoyage 需求文档》的基础上，给出首期 MVP 的技术架构、模块划分、接口契约、数据模型、安全设计与部署方案，作为开发、联调、测试与验收的技术依据。本文档中的选型与实现细节为技术设计方案，实际落地以开发阶段最终代码为准。

### 1.2 与需求文档的关系

- 需求文档定义“做什么”（业务边界、功能需求、验收标准）。
- 本文档定义“怎么做”（架构、技术栈、接口、数据流、部署）。
- 凡本文档与需求文档冲突之处，以需求文档的业务约束为准。

### 1.3 术语与缩略语

| 术语 | 说明 |
| --- | --- |
| LLM | 大语言模型，负责意图识别、参数抽取、答案组织 |
| MCP | Model Context Protocol，规范 LLM 与工具（数据库、API、爬虫）的连接 |
| A2A | Agent-to-Agent 协议，用于 Agent 间任务分派与结果汇总 |
| Orchestrator | 调度 Agent，负责意图路由与结果汇总 |
| Specialist Agent | 专业查询 Agent（天气、交通票务、演唱会） |
| MCP Tool | 通过 MCP 暴露的原子数据能力（查询、爬虫等） |

## 2. 总体架构

### 2.1 架构分层

系统采用分层 + 多 Agent 协作架构，自顶向下分为五层：

```
┌─────────────────────────────────────────────┐
│  接入层 (Gateway / API / Web UI)             │  用户输入、基础校验、安全网关
├─────────────────────────────────────────────┤
│  对话与编排层 (Conversation & Orchestration) │  会话管理、意图识别、参数抽取
├─────────────────────────────────────────────┤
│  Agent 协作层 (A2A)                          │  调度 Agent + 专业查询 Agent
├─────────────────────────────────────────────┤
│  工具接入层 (MCP Server)                     │  天气/火车/飞机/演唱会 工具
├─────────────────────────────────────────────┤
│  数据/外部依赖层 (Data & External)           │  DB（只读）、第三方 API、合规爬虫
└─────────────────────────────────────────────┘
```

## 3. 技术选型

> 以下为推荐选型，最终以开发环境可用性及课程要求为准。

| 维度 | 推荐方案 | 说明 |
| --- | --- | --- |
| 后端语言 | Python 3.11+ | 生态成熟，LLM/MCP SDK 支持完善 |
| Web 框架 | FastAPI | 异步、自动 OpenAPI 文档、易做接口契约 |
| LLM 接入 | OpenAI 兼容 SDK / 国产模型 API | 通过配置切换供应商，支持降级 |
| Agent 框架 | 自建轻量 Agent 调度 + A2A 消息协议 | MVP 仅保留必要 Agent，避免过度复杂 |
| MCP | 官方 MCP Python SDK | 工具注册、调用、超时标准化 |
| 数据存储 | SQLite / PostgreSQL（只读账号） | 演示数据缓存，最小权限只读 |
| 配置管理 | 环境变量 + `.env`（不入库） | 密钥注入，环境隔离 |
| 日志/观测 | 结构化日志（JSON）+ 请求 ID 链路追踪 | 满足可观测性需求 |
| 测试 | pytest + 契约测试 + 端到端 mock | 覆盖 AC-01~AC-14 |

## 4. 模块设计

### 4.1 接入网关 (gateway)

职责：
- 接收自然语言查询，生成唯一 `request_id`。
- 输入校验：非空、长度上限（如 512 字符）、基础安全检查。
- 基础限流与请求体大小限制。
- 统一错误包装，对外不暴露内部细节。

关键接口：
- `POST /api/v1/query`：提交查询，返回统一响应或参数补充提示。
- `GET /health`：健康检查（区分应用存活与依赖可用性）。

### 4.2 编排服务 (orchestrator service)

职责：
- 会话状态管理（会话内参数合并、单轮 + 一次补充）。
- 调用 LLM 完成意图识别与参数抽取（结构化输出）。
- 参数校验、相对日期转换（基于配置业务时区）、歧义确认。
- 将校验后的任务交给调度 Agent。

内部组件：
- `SessionStore`：内存或轻量存储，保留当前会话已确认参数。
- `IntentRecognizer`：LLM 封装，输出 `{intent, params, confidence, ambiguous}`。
- `ParamValidator`：按查询类型校验必要/可选参数。

### 4.3 调度 Agent (Orchestrator Agent)

职责（对应 FR-09 / FR-10）：
- 按意图路由到对应专业 Agent。
- 维护任务状态：待处理 / 处理中 / 成功 / 部分成功 / 失败 / 超时。
- 汇总多 Agent 结果，部分失败时保留有效结果并说明缺失。
- 超时控制，避免无限等待。

A2A 消息结构：

```json
{
  "request_id": "req-xxxx",
  "task_id": "task-xxxx",
  "task_type": "weather|train|flight|concert",
  "version": "1.0",
  "params": { "city": "北京", "date": "2025-08-11" },
  "deadline": "2025-08-11T10:00:00Z",
  "trace": { "parent": "req-xxxx", "span": "orchestrator" }
}
```

A2A 响应结构：

```json
{
  "task_id": "task-xxxx",
  "status": "success|partial|failed|timeout",
  "result": { ... },
  "source": "xxx",
  "updated_at": "2025-08-11T09:00:00Z",
  "error": { "code": "E_XXX", "message": "公开可说明信息" }
}
```

### 4.4 专业查询 Agent

| Agent | 处理意图 | 调用工具 |
| --- | --- | --- |
| WeatherAgent | 天气 | weather_tool |
| TransitAgent | 火车票、飞机票 | train_tool, flight_tool |
| ConcertAgent | 演唱会 | concert_tool |

每个 Agent 负责：参数二次校验 → 调用 MCP 工具 → 结果规范化 → 返回结构化结果。

### 4.5 MCP 工具层

每个工具声明（对应 FR-11 / FR-12）：
- `name` / `description`
- `input_schema`（必要 + 可选字段、类型）
- `output_schema`
- `timeout_ms`
- `error_types`

工具清单：

| 工具 | 数据源 | 输出核心字段 |
| --- | --- | --- |
| weather_tool | 天气 API / 模拟 | 城市、日期、天气状况、温度、降水、风力、来源、更新时间 |
| train_tool | 铁路 API / 模拟 | 车次、出发/到达站、时间、历时、席别、余票、来源 |
| flight_tool | 航班 API / 模拟 | 航班号、机场、起降时间、价格、舱位、来源 |
| concert_tool | 演出 API/爬虫 / 模拟 | 名称、艺人、日期、场馆、票价区间、状态、来源 |

SQL 安全（如引入 DB 工具）：
- 仅只读账号、白名单表/字段。
- 单条只读查询，禁止 DDL/DML/多语句/系统表。
- 执行前语法与操作类型校验、结果数量上限、查询超时。
- 优先参数化查询或受控模板；错误脱敏，不暴露凭据与堆栈。

## 5. 核心业务流程

以“天气查询”为例：

1. 用户 `POST /api/v1/query`：`北京 2025-08-11 天气怎么样`。
2. 网关生成 `request_id`，做输入校验。
3. 编排服务调用 LLM 识别意图=`weather`，抽取 `city=北京, date=2025-08-11`。
4. `ParamValidator` 校验通过。
5. 调度 Agent 派发 A2A 任务给 WeatherAgent。
6. WeatherAgent 通过 MCP 调用 `weather_tool`。
7. 工具返回结构化结果 + 来源 + 更新时间。
8. WeatherAgent 规范化结果，返回调度 Agent。
9. 调度 Agent 汇总，编排服务调用 LLM 组织用户可读回答（仅基于事实字段）。
10. 返回统一响应：条件摘要、结果、来源、查询时间、提示。

异常分支：参数缺失 → 列出缺失项；意图不明 → 请求澄清；工具超时 → 返回脱敏提示；部分失败 → 保留有效结果。

## 6. 数据模型

### 6.1 核心实体（逻辑模型）

| 实体 | 核心字段 |
| --- | --- |
| QueryRequest | request_id, session_id, raw_input, intent, params, created_at |
| QueryTask | task_id, request_id, agent_type, tool_type, status, started_at, ended_at |
| ToolResult | request_id, task_id, content, source, source_updated_at, fetched_at, status, error_code |
| UnifiedResponse | request_id, condition_summary, normalized_result, source, generated_at, hints |

### 6.2 任务状态枚举

`PENDING / PROCESSING / SUCCESS / PARTIAL / FAILED / TIMEOUT`

### 6.3 数据保留

- 仅保留排障/性能所需最少元数据。
- 不记录密码、API Key、Token、Cookie、Authorization 头。
- 保存周期可配置。

## 7. 统一响应规范

统一响应结构（面向用户）：

```json
{
  "request_id": "req-xxxx",
  "condition_summary": "北京 2025-08-11 天气",
  "results": [ { ... } ],
  "source": "天气数据提供方",
  "query_time": "2025-08-11T09:00:00+08:00",
  "hints": "数据仅供参考，具体以官方发布为准"
}
```

规则：
- 区分“无符合条件数据”与“服务不可用”。
- 价格字段必须含数值 + 币种。
- 模拟数据必须明确标记为“演示数据”。
- 缺失字段省略或标记“暂无数据”，不补造。

## 8. 安全设计

### 8.1 密钥与凭据
- 通过环境变量/安全配置注入，不写入源码、测试、文档。
- 数据库最小权限只读账号。

### 8.2 输入与输出不可信
- LLM 输出、工具返回值均校验。
- 对外 URL、SQL、日志、页面输出做注入与转义防护。
- 防止提示词注入泄露系统提示、凭据或越权。

### 8.3 敏感信息保护
- 用户响应、日志、异常中不出现 Key/Token/Cookie/凭据/认证头。
- SQL 错误脱敏，留审计记录（脱敏）。

### 8.4 接口防护
- 基础限流、请求体大小限制、超时控制。

## 9. 可观测性

- 日志字段：`request_id, task_id, agent_type, tool_type, status, duration_ms, error_code`。
- 指标：各查询类型调用次数、成功率、超时率、响应时间。
- 链路追踪：按 `request_id` 串联意图识别 → Agent 调度 → 工具调用。
- 健康检查区分应用存活与依赖可用性。

## 10. 部署方案

### 10.1 组件部署
- 后端服务（FastAPI）单进程/容器部署。
- MCP Server 可独立进程或同进程（MVP 建议同进程降低复杂度）。
- 数据库只读实例（或 SQLite 文件）。

### 10.2 环境隔离
- `dev` / `test` / `demo` 配置隔离，使用独立 `.env`。
- 第三方 Key 仅 demo 环境使用模拟工具时可缺省。

### 10.3 性能目标
- 单类查询端到端目标 < 10s。
- 每工具独立超时（如 5s）。
- 单次返回结果默认 ≤ 20 条。

## 11. 测试方案

### 11.1 测试分层
- 单元测试：意图识别、参数校验、日期转换、SQL 安全校验。
- 契约测试：A2A 消息、MCP 工具输入输出。
- 端到端测试：用可控 mock 工具覆盖 AC-01 ~ AC-14。

### 11.2 必覆盖场景（每类查询）
正常 / 参数缺失 / 无结果 / 数据格式异常 / 工具超时。

### 11.3 专项测试
- SQL 只读限制（AC-11）。
- 提示词注入防护。
- 敏感信息脱敏（AC-12）。
- 结果真实性（AC-13，字段可追溯）。
- 链路追踪（AC-14）。

## 12. 开发里程碑

### P0（必须）
- 接入网关 + 统一查询接口。
- 四类意图识别与参数抽取。
- 参数校验与补充询问。
- 调度 + 专业 Agent 基础 A2A 协作。
- MCP 工具注册、调用、超时、错误处理。
- 每类查询至少一个可运行适配器或标记模拟工具。
- 统一响应 + 来源/查询时间。
- SQL 只读控制、敏感信息保护、链路日志。
- 核心自动化测试与演示流程。

### P1（稳定性增强）
- 会话内条件修改。
- 部分失败结果返回。
- 结果排序/筛选/数量限制。
- 指标统计、依赖健康检查、完整契约测试。
- 授权允许时替换真实数据源。

## 14. 附录：错误码规范（建议）

| 错误码 | 含义 | 用户提示 |
| --- | --- | --- |
| E_PARAM_MISSING | 必要参数缺失 | 列出缺失参数并引导补充 |
| E_INTENT_UNKNOWN | 意图无法识别 | 请求澄清查询目标 |
| E_TOOL_TIMEOUT | 工具超时 | 服务处理超时，请稍后重试 |
| E_TOOL_UNAVAILABLE | 数据源不可用 | 服务暂时不可用，请稍后重试 |
| E_NO_RESULT | 无符合条件数据 | 明确无结果，非系统异常 |
| E_INVALID_INPUT | 输入非法 | 给出可操作修正提示 |
| E_SQL_REJECTED | SQL 被安全策略拦截 | 内部脱敏记录，不对用户暴露细节 |
